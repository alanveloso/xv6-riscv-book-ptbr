\chapter{Organização do sistema operacional}
\label{CH:FIRST}

Um requisito fundamental para um sistema operacional é suportar diversas atividades ao mesmo tempo. Por exemplo, usando a interface de chamada de sistema %ou é melhor deixar "system call" em inglês mesmo? 
descrita no Capítulo~\ref{CH:UNIX}
um processo pode iniciar novos processos com 
\lstinline{fork}.
O sistema operacional deve 
\indextext{compartilhar (time-share)} 
os recursos do computador entre esses processos.
Por exemplo, mesmo que haja mais processos
do que CPUs de hardware, o sistema operacional
deve garantir que todos os processos
tenham a chance de ser executados.  O sistema operacional também deve providenciar
\indextext{isolamento} 
entre os processos.
Ou seja, se um processo tiver um bug e apresentar mau funcionamento, ele não deve afetar os processos que não dependem do processo com bug. O isolamento completo, entretanto, é muito forte, pois deve ser possível que os processos interajam intencionalmente; os pipelines são um exemplo.
Portanto, um sistema operacional deve atender a três requisitos: multiplexação, isolamento,
e interação.

Este capítulo apresenta uma visão geral de como os sistemas operacionais são
organizados para atender a esses três requisitos. Existem muitas maneiras de
fazer isso, mas este texto se concentra nos projetos mais comuns
centrados em um \indextext{kernel monolítico}, que é usado por muitos sistemas
operacionais Unix. Este capítulo também fornece uma visão geral de um processo
xv6, que é a unidade de isolamento no xv6, e a criação
do primeiro processo quando o xv6 é iniciado.

Xv6 é executado em um \indextext{multi-core}\footnote{
Por ``multi-core'' este texto significa várias CPUs que compartilham memória,
mas executam em paralelo, cada uma com seu próprio conjunto de registros.
Este texto às vezes usa o termo
\indextext{multiprocessador}  como sinônimo de multi-core,
embora multiprocessador também possa se referir mais especificamente a um computador com
vários chips de processador distintos.} microprocessador RISC-V,
e grande parte de sua funcionalidade de baixo nível
(por exemplo, sua implementação de processo) é específica do
RISC-V. O RISC-V é uma CPU de 64 bits, e o xv6 é escrito em C ``LP64'',
o que significa que long (L) e ponteiros (P) na linguagem de programação C
são de 64 bits, mas um {\tt int} é de 32 bits. Este livro pressupõe que o 
leitor tenha feito um pouco de programação em nível de máquina em alguma 
arquitetura e introduzirá ideias específicas do RISC-V à medida que forem surgindo.
Os documentos ISA em nível de usuário~\cite{riscv:user} e arquitetura
privilegiada~\cite{riscv:priv} são as especificações completas.
Você também pode consultar
``The RISC-V Reader: An Open Architecture Atlas''~\cite{riscv}.

Em um computador completo, a CPU é cercada por hardware de suporte, sendo que
grande parte dele está na forma de interfaces de E/S. O Xv6 foi escrito 
para o hardware de suporte simulado pela opção ``-machine virt'' do qemu. Isso inclui
RAM, uma ROM contendo código de inicialização, uma conexão serial com o teclado/tela
do usuário e um disco para armazenamento.

\section{Abstração de recursos físicos}
%% 

A primeira pergunta que alguém pode fazer ao se deparar com um sistema operacional é:
por que tê-lo? Ou seja, é possível implementar as chamadas de sistema na
Figura~\ref{fig:api}
como uma biblioteca, com a qual as aplicações se ligam. Nesse sentido,
cada aplicação poderia até mesmo ter sua própria biblioteca adaptada às suas necessidades.
As aplicações poderiam interagir diretamente com os recursos de hardware
e usar esses recursos da melhor maneira para a aplicação (por exemplo, para obter um desempenho alto ou previsível). Alguns sistemas operacionais para
sistemas embarcados ou sistemas em tempo real são organizados dessa forma.

A desvantagem dessa abordagem de biblioteca é que, se houver mais de um aplicativo
em execução, os aplicativos deverão ser bem comportados.
Por exemplo, cada aplicativo deve ceder periodicamente a CPU
para que outros aplicativos possam ser executados.
Esse esquema de compartilhamento de tempo 
\textit{cooperativo} 
pode ser adequado se todos os aplicativos confiarem uns nos outros e não tiverem bugs. É mais comum que os aplicativos não confiem uns nos outros e tenham bugs, portanto, muitas vezes é necessário um isolamento mais forte do que o fornecido por um esquema cooperativo.

Para obter um isolamento forte, é útil proibir que aplicativos acessem diretamente recursos de hardware sensíveis e, em vez disso, abstrair os recursos em serviços. Por exemplo, aplicativos Unix interagem com o armazenamento apenas por meio das chamadas de sistema
\lstinline{open},
\lstinline{read},
\lstinline{write}, 
e
\lstinline{close}
do sistema de arquivos,
em vez de ler e gravar diretamente no disco. 
Isso proporciona ao aplicativo a conveniência dos nomes de caminho e permite que
o sistema operacional (como implementador da interface) gerencie o disco. 
Mesmo que o isolamento não seja uma preocupação, é provável que os programas
que interagem intencionalmente (ou apenas desejam manter
fora do caminho uns dos outros) considerem o sistema de arquivos uma abstração
mais conveniente do que o uso direto do disco.

Da mesma forma, o Unix alterna de forma transparente as CPUs de hardware entre os processos,
salvando e restaurando o estado dos registradores conforme necessário,
para que os aplicativos não precisem se preocupar com o compartilhamento de tempo. 
Essa transparência permite que o sistema operacional compartilhe
CPUs mesmo que alguns aplicativos estejam em loops infinitos.

Como outro exemplo, os processos Unix usam 
\lstinline{exec}
para construir sua imagem de memória, em vez de interagir diretamente com a memória física. 
Isso permite que o sistema operacional decida onde colocar um processo na memória; 
se a memória for escassa, o sistema operacional poderá até mesmo armazenar alguns dos
dados de um processo no disco.
\lstinline{exec}
também oferece aos usuários a conveniência de um sistema de arquivos
para armazenar imagens de programas executáveis.  

Muitas formas de interação entre os processos do Unix ocorrem por meio de descritores de arquivos.
Os descritores de arquivos não apenas abstraem muitos detalhes
(por exemplo, onde os dados em um canal ou arquivo são armazenados), 
mas também são definidos de uma maneira que simplifica a interação.
Por exemplo, se um aplicativo em um pipeline falhar, o kernel
gera um sinal de fim de arquivo para o próximo processo no pipeline.

A interface de chamada de sistema na Figura ~\ref{fig:api}
foi cuidadosamente projetada para oferecer conveniência ao programador e
a possibilidade de forte isolamento. A interface Unix
não é a única maneira de abstrair recursos, mas provou ser uma boa
maneira.

\section{Modo de usuário, modo de supervisor e chamadas de sistema}
%% 

O isolamento forte requer um limite rígido entre as aplicações e o sistema operacional. 
Se a aplicação cometer um erro, não queremos que o sistema operacional
falhe ou que outras aplicações falhem. Em vez disso, o sistema operacional deve ser
capaz de limpar a aplicação com falha e continuar executando outros aplicativos.
Para obter um isolamento forte, o sistema operacional deve providenciar para que as aplicações não possam modificar (ou mesmo ler) as estruturas de dados e instruções do sistema operacional
e que os aplicativos não possam acessar a memória de outros processos.

As CPUs oferecem suporte de hardware para um forte isolamento.
Por exemplo, o RISC-V tem três modos nos quais a CPU pode executar instruções:
\indextext{modo de máquina},
\indextext{modo supervisor} e
\indextext{modo usuário}.
As instruções executadas no modo de máquina têm privilégio total; uma CPU
é iniciada no modo de máquina. O modo de máquina destina-se principalmente a
configurar o computador durante a inicialização.
O Xv6 executa algumas linhas no modo de máquina e depois muda para o modo supervisor.

No modo supervisor, a CPU tem permissão para executar 
\indextext{instruções privilegiadas}:
por exemplo, ativar e desativar interrupções, ler e gravar
o registro que contém o endereço de uma tabela de páginas etc.
Se um aplicativo no modo de usuário tentar executar
uma instrução privilegiada, a CPU não executará a instrução, mas mudará
para o modo de supervisor para que o código do modo de supervisor possa encerrar o aplicativo,
porque ele fez algo que não deveria fazer.
Figura~\ref{fig:os}
no Capitulo~\ref{CH:UNIX} ilustra essa organização. Uma aplicação pode
executar somente instruções no modo de usuário (por exemplo, somar números etc.) e é considerado
em execução no 
\indextext{modo usuário},
enquanto o software no modo supervisor também pode executar instruções privilegiadas e
é considerado em execução no
\indextext{espaço do kernel}.
O software executado no espaço do kernel (ou no modo supervisor) é chamado de
\indextext{kernel}.

Um aplicativo que deseja invocar uma função do kernel (por exemplo, a chamada do sistema
\lstinline{read}
no xv6) deve 
fazer a transição para o kernel; um aplicativo \emph{não pode} invocar uma função do kernel
diretamente. As CPUs fornecem uma instrução especial que alterna a CPU
do modo de usuário para o modo de supervisor e entra no kernel em um ponto de entrada
especificado pelo kernel. (O RISC-V
fornece a instrução 
\indexcode{ecall}
para essa finalidade.) Depois que a CPU passa para o modo supervisor,
o kernel pode validar os argumentos da chamada do sistema (por exemplo,
verificar se o endereço passado para a chamada do sistema faz parte da memória do aplicativo), decidir se o aplicativo tem permissão para executar a operação solicitada (por exemplo,
verificar se o aplicativo tem permissão para gravar o arquivo especificado)
e, em seguida, negá-la ou executá-la. É importante que o kernel controle o ponto de entrada para
transições para o modo supervisor; se o aplicativo pudesse decidir o ponto de entrada do kernel, um aplicativo mal-intencionado poderia, por exemplo, entrar no kernel em um ponto em que a validação dos argumentos fosse ignorada.

\section{Organização do Kernel}
%% 

Uma questão fundamental do projeto é qual parte do sistema operacional
deve ser executada no modo supervisor. 
Uma possibilidade é que todo o sistema operacional resida
no kernel, de modo que as implementações de todas as chamadas do sistema
sejam executadas no modo supervisor.
Essa organização é chamada de
\indextext{Núcleo monolítico}.

Nessa organização, todo o sistema operacional consiste em um único programa
executado com privilégio total de hardware
. Essa organização é conveniente porque o projetista do sistema operacional não tem
que decidir quais partes do sistema operacional não precisam de privilégio total de hardware
. Além disso, é mais fácil para as diferentes partes do sistema operacional
cooperar.  Por exemplo, um sistema operacional pode ter um cache de buffer que pode
ser compartilhado pelo sistema de arquivos e pelo sistema de memória virtual.

Uma desvantagem da organização monolítica é que as interações entre
as diferentes partes do sistema operacional são muitas vezes complexas (como veremos no
restante deste texto) e, portanto, é fácil para um desenvolvedor de sistema operacional
cometer um erro.  Em um kernel monolítico, um erro é fatal, pois um erro
no modo supervisor geralmente faz com que o kernel falhe.  Se o kernel falhar,
o computador para de funcionar e, portanto, todos os aplicativos também falham.  O computador
precisa ser reinicializado para começar novamente.

Para reduzir o risco de erros no kernel, os projetistas de sistemas operacionais podem minimizar a
quantidade de código do sistema operacional que é executada no modo supervisor e executar a
maior parte do sistema operacional no modo de usuário.
Essa organização do kernel é chamada de
\indextext{microkernel}.

\begin{figure}[t]
\center
\includegraphics[scale=0.5]{fig/mkernel.pdf}
\caption{Um microkernel com um servidor de sistema de arquivos}
\label{fig:mkernel}
\end{figure}

Figura~\ref{fig:mkernel}
ilustra esse design de microkernel. Na figura, o sistema de arquivos é executado como um processo de nível de usuário. 
Os serviços do sistema operacional executados como processos são chamados de servidores.
Para permitir que os aplicativos interajam com o servidor de arquivos, o kernel fornece um mecanismo de comunicação entre processos
para enviar mensagens de um processo em modo de usuário
para outro. Por exemplo, se um aplicativo como o shell quiser ler ou gravar um arquivo,
ele enviará uma mensagem ao servidor de arquivos e aguardará por uma resposta.

Em um microkernel, a interface do kernel consiste em algumas funções de baixo nível
para iniciar aplicativos, enviar mensagens,
acessar o hardware do dispositivo, etc.  Essa organização permite que o kernel seja 
relativamente simples, pois a maior parte do sistema operacional
reside em servidores de nível de usuário.

No mundo real, tanto os kernels monolíticos quanto os microkernels são
populares. Muitos kernels do Unix são monolíticos. 
Por exemplo, o Linux tem um kernel monolítico,
embora algumas funções do sistema operacional sejam executadas como servidores de nível de usuário (por exemplo, o sistema de janelas).
O Linux oferece alto desempenho para aplicativos com uso intensivo do sistema operacional, em parte porque os subsistemas do kernel podem ser fortemente integrados.

Sistemas operacionais como Minix, L4 e QNX são organizados como um microkernel com servidores
e têm sido amplamente implantados em configurações incorporadas. Uma variante do
do L4, o seL4, é pequena o suficiente para ter sido verificada quanto à segurança da memória
e outras propriedades de segurança~\cite{sel4}.

Há muito debate entre os desenvolvedores de sistemas operacionais sobre qual
organização é melhor, e não há evidências conclusivas de uma forma ou
de outra.  Além disso, depende muito do que significa “melhor”:
desempenho mais rápido, tamanho de código menor, confiabilidade do kernel,
confiabilidade do sistema operacional completo (incluindo serviços de nível de usuário), etc.

Há também considerações práticas que podem ser mais importantes
do que a questão de qual organização.  Alguns sistemas operacionais
têm um microkernel, mas executam alguns dos serviços de nível de usuário no espaço do kernel
por motivos de desempenho. Alguns sistemas operacionais têm kernels
monolíticos porque foi assim que começaram e há pouco incentivo
para mudar para uma organização de microkernel puro, pois os novos recursos podem
ser mais importantes do que reescrever o sistema operacional existente para se adequar a um design de microkernel.

Do ponto de vista deste livro, os sistemas operacionais monolíticos e de microkernel
compartilham muitas ideias importantes.  Eles implementam chamadas de sistema, usam
tabelas de páginas, lidam com interrupções, oferecem suporte a processos, usam
bloqueios para controle de concorrência, implementam um sistema de arquivos,
etc. Este livro se concentra nessas ideias centrais.

O Xv6 é implementado como um kernel monolítico, como a maioria dos sistemas operacionais Unix.
Assim, a interface do kernel do xv6 corresponde à interface do sistema operacional, 
e o kernel implementa o sistema operacional completo. Como o xv6 não fornece muitos serviços, 
seu kernel é menor do que alguns microkernels, mas conceitualmente o xv6 é monolítico.

\section{Visão geral do processo}
%% 

A unidade de isolamento no xv6 (como em outros sistemas operacionais Unix) é um 
\indextext{processo}.
A abstração do processo impede que um processo destrua ou espione a memória, a CPU, os descritores de arquivos etc. de outro processo.  Ela também impede que um processo
destrua o próprio kernel, de modo que um processo não possa subverter os mecanismos de isolamento do kernel.
O kernel deve implementar a abstração do processo com cuidado porque
um aplicativo com bugs ou mal-intencionado pode enganar o kernel ou o hardware para fazer
algo ruim (por exemplo, contornar o isolamento). Os mecanismos usados pelo
kernel para implementar processos incluem o sinalizador de modo de usuário/supervisor, espaços de endereço, e divisão de tempo de threads.

Para ajudar a reforçar o isolamento, a abstração do processo fornece a
ilusão a um programa de que ele tem sua própria máquina privada.  Um processo fornece
a um programa o que parece ser um sistema de memória privado, ou
\indextext{espaço de endereço}, 
que outros processos não podem ler ou gravar.
Um processo também fornece ao programa o que parece ser sua própria CPU
para executar as instruções do programa.

O Xv6 usa tabelas de páginas (que são implementadas por hardware) para dar a cada processo
seu próprio espaço de endereço. A tabela de páginas do RISC-V
traduz (ou ``mapeia'') um
\indextext{endereço virtual}
(o endereço que uma instrução RISC-V manipula) para um
\indextext{endereço físico}
(um endereço que a CPU envia para a memória principal).

\begin{figure}[t]
\centering
\includegraphics[scale=0.5]{fig/as.pdf}
\caption{Layout do espaço de endereço virtual de um processo}
\label{fig:as}
\end{figure}

O Xv6 mantém uma tabela de páginas separada para cada processo que define o espaço de endereço
desse processo. Conforme ilustrado na 
Figura~\ref{fig:as},
um espaço de endereço inclui a
\indextext{user memory}
do processo, começando no endereço virtual zero. As instruções vêm primeiro,
seguidas pelas variáveis globais, depois a pilha,
e, finalmente, uma área de ``heap'' (para malloc)
que o processo pode expandir conforme necessário.
Há vários fatores que limitam o
tamanho máximo do espaço de endereço de um processo:
ponteiros no RISC-V têm 64 bits de largura;
o hardware usa somente os 39 bits mais baixos quando
procura endereços virtuais nas tabelas de páginas;
e xv6 usam somente 38 desses 39 bits.
Portanto, o endereço máximo é $2^{38}-1$ =
No topo do espaço de endereço, o xv6 coloca
uma página \indextext{trampoline} page (4096 bytes)
e uma página \indextext{trapframe}.  O Xv6 usa essas duas páginas
para fazer a transição para o kernel e vice-versa;
a página trampoline contém o código para fazer a transição para dentro e para fora
do kernel, e o trapframe é onde o kernel salva
os registros de usuário do processo,
como explica o Capítulo~\ref{CH:TRAP}.

O kernel xv6 mantém muitas partes de estado para cada processo,
que ele reúne em um
\indexcode{struct proc}
As partes mais importantes do estado do kernel de um processo são sua
tabela de páginas, sua pilha do kernel e seu estado de execução.
Usaremos a notação
\indexcode{p->xxx}
para se referir a elementos da
\lstinline{proc}
estrutura; por exemplo,
\indexcode{p->pagetable} é um ponteiro para a tabela de páginas do processo.

Cada processo possui uma thread de controle (ou
\indextext{thread}
para abreviar) que contém o estado necessário para executar o processo.
A qualquer momento, uma thread pode estar em execução na CPU,
ou suspensa (sem execução, mas capaz de retomar a execução
no futuro).
Para alternar uma CPU entre processos,
o kernel suspende a thread em execução naquela CPU
e salva seu estado, e restaura o estado da thread suspensa anteriormente de outro processo. 
Grande parte do estado de uma thread (variáveis locais, endereços de retorno de chamadas de função) é armazenada nas pilhas da thread.
Cada processo possui duas pilhas: uma pilha do usuário e uma pilha do kernel
(\indexcode{p->kstack}).
Quando o processo está executando instruções do usuário, apenas sua pilha do usuário
está em uso e sua pilha do kernel está vazia.
Quando o processo entra no kernel (para uma chamada de sistema ou interrupção),
o código do kernel é executado na pilha do kernel do processo; enquanto
um processo está no kernel, sua pilha do usuário ainda contém dados
salvos, mas não é usada ativamente.
A thread de um processo alterna entre usar ativamente sua pilha do usuário
e sua pilha do kernel. A pilha do kernel é separada (e protegida do
código do usuário) para que o kernel
possa ser executado mesmo que um processo tenha destruído sua pilha do usuário.

Um processo pode fazer uma chamada de sistema executando a instrução RISC-V \indexcode{ecall}. Esta instrução aumenta o nível de privilégio de hardware e
altera o contador de programa para um ponto de entrada definido pelo kernel. O código
no ponto de entrada alterna para a pilha do kernel do processo e executa as instruções do kernel
que implementam a chamada de sistema. Quando a chamada de sistema
é concluída, o kernel retorna para a pilha do usuário e retorna ao
espaço do usuário chamando a instrução \indexcode{sret}, que reduz
o nível de privilégio de hardware e retoma a execução das instruções do usuário
logo após a instrução de chamada de sistema. A thread de um processo pode
``bloquear'' no kernel para aguardar a E/S e retomar de onde parou
quando a E/S terminar.

\indexcode{p->state} 
indica se o processo está alocado, pronto
para ser executado, atualmente em execução em uma CPU, aguardando E/S ou saindo.

\indexcode{p->pagetable}
contém a tabela de páginas do processo, no formato
esperado pelo hardware RISC-V.
O Xv6 faz com que o hardware de paginação utilize a
\lstinline{p->pagetable}
de um processo ao executá-lo no espaço do usuário.
A tabela de páginas de um processo também serve como registro dos
endereços das páginas físicas alocadas para armazenar a
memória do processo.

Em resumo, um processo agrupa duas ideias de design: um espaço de endereço para
dar a um processo a ilusão de sua própria memória e uma thread para
dar ao processo a ilusão de sua própria CPU. No xv6, um processo consiste
em um espaço de endereço e uma thread. Em sistemas operacionais reais,
um processo pode ter mais de uma thread para aproveitar múltiplas CPUs.

\section{Código: iniciando xv6, o primeiro processo e chamada de sistema}

Para tornar o xv6 mais concreto, descreveremos como o kernel inicia e

\section{Modelo de Segurança}

Você pode se perguntar como o sistema operacional lida com códigos maliciosos ou com bugs.
Como lidar com códigos maliciosos é mais difícil do que lidar com bugs acidentais, é razoável
concentrar-se principalmente em fornecer segurança contra malware.
Aqui está uma visão geral das premissas e
objetivos típicos de segurança no design de sistemas operacionais.

O sistema operacional deve assumir que o código em nível de usuário de um processo fará o possível para destruir o kernel ou outros processos. O código do usuário pode tentar desreferenciar ponteiros fora de seu espaço de endereço permitido; pode tentar executar quaisquer instruções RISC-V, mesmo aquelas não destinadas ao código do usuário; pode tentar ler e escrever em qualquer registrador de controle RISC-V; pode tentar acessar diretamente o hardware do dispositivo; e pode passar valores inteligentes para chamadas de sistema na tentativa de enganar o kernel, fazendo-o travar ou fazer algo estúpido. O objetivo do kernel é restringir cada processo do usuário para que tudo o que ele possa fazer seja ler/escrever/executar sua própria memória de usuário, usar os 32 registradores RISC-V de uso geral e afetar o kernel e outros processos da maneira que as chamadas de sistema devem permitir. O kernel deve impedir quaisquer outras ações. Isso normalmente é um requisito absoluto no projeto do kernel.

As expectativas para o código do kernel são bem diferentes. O código do kernel
é assumido como escrito por programadores bem-intencionados e cuidadosos.
Espera-se que o código do kernel esteja livre de bugs e certamente não contenha
nada malicioso. Essa suposição afeta a forma como analisamos o código do kernel.
Por exemplo, existem muitas funções internas do kernel (por exemplo, os bloqueios de rotação)
que causariam sérios problemas se o código do kernel as utilizasse
incorretamente. Ao examinar qualquer trecho específico do código do kernel,
queremos nos convencer de que ele se comporta corretamente. Assumimos,
no entanto, que o código do kernel em geral está escrito corretamente e segue
todas as regras sobre o uso das funções e estruturas de dados do kernel.
No nível de hardware, presume-se que a CPU, a RAM, o disco, etc. do RISC-V operem conforme anunciado na documentação, sem bugs de hardware.

É claro que na vida real as coisas não são tão simples. É
difícil impedir que um código de usuário inteligente torne um sistema inutilizável
(ou o cause em pânico) consumindo recursos protegidos pelo kernel —
espaço em disco, tempo de CPU, slots na tabela de processos, etc. Geralmente é
impossível escrever código de kernel livre de bugs ou projetar hardware livre de bugs; se os
autores de código de usuário malicioso estiverem cientes de bugs no kernel ou no hardware,
eles os explorarão. Mesmo em kernels maduros e amplamente utilizados, como
o Linux, as pessoas descobrem novas vulnerabilidades
continuamente~\cite{mitre:cves}. Vale a pena projetar salvaguardas no
kernel contra a possibilidade de haver bugs: asserções, verificação de
tipos, páginas de proteção de pilha, etc. Por fim, a distinção entre
código do usuário e do kernel às vezes é confusa: alguns processos privilegiados de nível de usuário
podem fornecer serviços essenciais e efetivamente fazer parte
do sistema operacional e, em alguns sistemas operacionais, o código privilegiado do usuário
pode inserir novo código no kernel (como com os módulos carregáveis
do kernel do Linux).

Você pode se perguntar como o sistema operacional lida com código com bugs ou malicioso. Como lidar com código malicioso é mais difícil do que com bugs acidentais, é razoável focar principalmente em fornecer segurança contra ações maliciosas. Aqui está uma visão geral das suposições e objetivos típicos de segurança no projeto de sistemas operacionais.

O sistema operacional deve assumir que o código de nível de usuário de um processo fará o possível para danificar o kernel ou outros processos. O código de usuário pode tentar desreferenciar ponteiros fora de seu espaço de endereçamento permitido; pode tentar executar qualquer instrução RISC-V, mesmo aquelas não destinadas ao código de usuário; pode tentar ler e escrever em qualquer registrador de controle do RISC-V; pode tentar acessar diretamente o hardware dos dispositivos; e pode passar valores engenhosos para chamadas de sistema tentando enganar o kernel para travar ou fazer algo incorreto. O objetivo do kernel é restringir cada processo de usuário para que ele só possa ler/escrever/executar sua própria memória de usuário, usar os 32 registradores de propósito geral do RISC-V, e afetar o kernel e outros processos apenas nas formas previstas pelas chamadas de sistema. O kernel deve impedir qualquer outra ação. Isso é tipicamente um requisito absoluto no projeto de kernels.

As expectativas para o próprio código do kernel são bastante diferentes. Assume-se que o código do kernel seja escrito por programadores bem-intencionados e cuidadosos. Espera-se que esse código esteja livre de bugs, e certamente que não contenha nada malicioso. Essa suposição afeta como analisamos o código do kernel. Por exemplo, há muitas funções internas do kernel (como os spinlocks) que causariam sérios problemas se usadas incorretamente. Ao examinar qualquer trecho específico de código do kernel, queremos nos convencer de que ele se comporta corretamente. Assumimos, no entanto, que o código do kernel em geral é escrito corretamente e segue todas as regras de uso das funções e estruturas de dados do próprio kernel. Em nível de hardware, assume-se que a CPU RISC-V, RAM, disco, etc. operem conforme descrito na documentação, sem bugs de hardware.

Claro que na vida real as coisas não são tão simples. É difícil impedir que código de usuário engenhoso torne o sistema inutilizável (ou leve a um pânico) ao consumir recursos protegidos pelo kernel — espaço em disco, tempo de CPU, slots na tabela de processos, etc. É geralmente impossível escrever um kernel sem bugs ou projetar hardware sem defeitos; se os autores de código malicioso conhecerem bugs no kernel ou no hardware, irão explorá-los. Mesmo em kernels maduros e amplamente utilizados, como o Linux, vulnerabilidades continuam sendo descobertas regularmente~\cite{mitre:cves}. Vale a pena projetar salvaguardas no kernel contra a possibilidade de ele conter bugs: afirmações (assertions), verificação de tipos, páginas de guarda na pilha, etc. Finalmente, a distinção entre código de usuário e de kernel às vezes é difusa: alguns processos privilegiados em nível de usuário podem fornecer serviços essenciais e funcionar efetivamente como parte do sistema operacional, e em alguns sistemas operacionais o código de usuário privilegiado pode inserir novo código no kernel (como nos módulos carregáveis do kernel no Linux).

\section{Mundo real}
%% 

A maioria dos sistemas operacionais adotou o conceito de processo, e a maioria
dos processos se parece com os do xv6. Os sistemas operacionais modernos, no entanto,
suportam várias threads dentro de um processo, permitindo que um único processo
explore múltiplas CPUs. Suportar múltiplas threads em um
processo envolve uma série de mecanismos que o xv6 não possui,
geralmente incluindo mudanças na interface (por exemplo, o
\lstinline{clone} do Linux,
uma variante do
\lstinline{fork}),
para controlar quais aspectos de
um processo as threads compartilham.
