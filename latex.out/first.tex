\chapter{Organização do sistema operacional}
\label{CH:FIRST}

Um requisito fundamental para um sistema operacional é suportar várias atividades simultaneamente. Usando a interface de chamada de sistema descrita no Capítulo~\ref{CH:UNIX} um processo pode iniciar novos processos com \lstinline{fork}.O sistema operacional deve\indextext{compartilharo tempo} dos recursos do computador entre esses processos. Por exemplo, mesmo que haja mais processos do que CPUs físicas, o sistema operacional deve garantir que todos os processos tenham a chance de executar. O sistema operacional também deve providenciar
\indextext{isolamento}entre os processos. Ou seja, se um processo tiver um bug e falhar, isso não deve afetar os processos que não dependem do processo com problema. No entanto, o isolamento completo é muito forte, já que deve ser possível para os processos interagirem intencionalmente; pipelines são um exemplo. Assim, um sistema operacional deve cumprir três requisitos: multiplexação, isolamento e interação.

Este capítulo fornece uma visão geral de como os sistemas operacionais são organizados para alcançar esses três requisitos. Existem muitas maneiras de fazer isso, mas este texto foca em designs convencionais centrados em um\indextext{kernel monolítico}, que é usado por muitos sistemas operacionais Unix. Este capítulo também fornece uma visão geral de um processo xv6, que é a unidade de isolamento no xv6, e a criação do primeiro processo quando o xv6 inicia.

O xv6 roda em um microprocessador \indextext{multi-core}\footnote{
Por multi-core'' este texto se refere a múltiplas CPUs que compartilham memória, mas executam em paralelo, cada uma com seu próprio conjunto de registradores. Este texto às vezes usa o termo
\indextext{multiprocessadorcomo sinônimo de multi-core, embora multiprocessador também possa se referir mais especificamente a um computador com vários chips de processador distintos.}, e grande parte de sua funcionalidade de baixo nível (por exemplo, sua implementação de processos) é específica para RISC-V. O RISC-V é uma CPU de 64 bits, e o xv6 é escrito em C LP64'', o que significa que longos (L) e ponteiros (P) na linguagem de programação C têm 64 bits, mas um {\tt inté de 32 bits. Este livro assume que o leitor fez um pouco de programação em nível de máquina em alguma arquitetura, e introduzirá ideias específicas do RISC-V à medida que surgirem.
 Os documentos da ISA de nível de usuário~\cite{riscv:user} e da arquitetura privilegiada~\cite{riscv:priv} são as especificações completas.
te{riscv:user} e da arquitetura privilegiada~\cite{riscv:priv} são as especificações completas. Você também pode se referir a ``The RISC-V Reader: An Open Architecture Atlas''~\cite{riscv}.

A CPU em um computador completo é cercada por hardware de suporte, muito dele na forma de interfaces de E/S. O xv6 é escrito para o hardware de suporte simulado pela opção ``-machine virt'' do qemu. Isso inclui RAM, um ROM contendo código de inicialização, uma conexão serial com o teclado/tela do usuário e um disco para armazenamento.

\section{Abstraindo Recursos Físicos}
%% 

A primeira pergunta que alguém pode fazer ao encontrar um sistema operacional é por que tê-lo? Ou seja, poderia-se implementar as chamadas de sistema na Figura~\ref{fig:api} como uma biblioteca, com a qual os aplicativos se vinculam. Nesse plano, cada aplicativo poderia até ter sua própria biblioteca adaptada às suas necessidades. Os aplicativos poderiam interagir diretamente com os recursos de hardware e usar esses recursos da melhor maneira para o aplicativo (por exemplo, para atingir alto ou previsível desempenho). Alguns sistemas operacionais para dispositivos embarcados ou sistemas em tempo real são organizados dessa maneira.

A desvantagem dessa abordagem de biblioteca é que, se houver mais de um aplicativo em execução, os aplicativos devem ser bem comportados. Por exemplo, cada aplicativo deve periodicamente ceder a CPU para que outros aplicativos possam rodar. Um esquema de compartilhamento de tempo
\textit{cooperativo} pode ser aceitável se todos os aplicativos confiarem uns nos outros e não tiverem bugs. É mais típico que os aplicativos não confiem uns nos outros e tenham bugs, então muitas vezes se deseja um isolamento mais forte do que um esquema cooperativo fornece.

Para alcançar um forte isolamento, é útil proibir os aplicativos de acessarem diretamente recursos de hardware sensíveis e, em vez disso, abstrair os recursos em serviços. Por exemplo, os aplicativos Unix interagem com armazenamento apenas por meio das chamadas de sistema 
\lstinline{open},
\lstinline{read},
\lstinline{write}, 
e
\lstinline{close}
sistemas de arquivo, em vez de ler e escrever diretamente no disco.
Isso fornece ao aplicativo a conveniência de nomes de caminho e permite que o sistema operacional (como o implementador da interface) gerencie o disco. Mesmo que o isolamento não seja uma preocupação, programas que interagem intencionalmente (ou que apenas desejam se manter afastados uns dos outros) provavelmente acharão um sistema de arquivos uma abstração mais conveniente do que o uso direto do disco.

De maneira semelhante, o Unix alterna de forma transparente as CPUs de hardware entre processos, salvando e restaurando o estado do registrador conforme necessário, para que os aplicativos não precisem estar cientes do compartilhamento de tempo. Essa transparência permite que o sistema operacional compartilhe CPUs mesmo que alguns aplicativos estejam em loops infinitos.

Como outro exemplo, os processos Unix usam
\lstinline{exec}
para construir sua imagem de memória, em vez de interagir diretamente com a memória física. Isso permite que o sistema operacional decida onde colocar um processo na memória; se a memória estiver apertada, o sistema operacional pode até armazenar alguns dados de um processo no disco.
\lstinline{exec}
também oferece aos usuários a conveniência de um sistema de arquivos para armazenar imagens de programas executáveis.

Muitas formas de interação entre processos Unix ocorrem por meio de descritores de arquivo. Os descritores de arquivo não apenas abstraem muitos detalhes (por exemplo, onde os dados em um pipe ou arquivo são armazenados), mas também são definidos de uma forma que simplifica a interação. Por exemplo, se um aplicativo em um pipeline falhar, o kernel gera um sinal de fim de arquivo para o próximo processo no pipeline.

A interface de chamada de sistema na Figura~\ref{fig:api}
é cuidadosamente projetada para fornecer tanto conveniência ao programador quanto a possibilidade de um forte isolamento. A interface Unix não é a única maneira de abstrair recursos, mas provou ser uma boa abordagem.

\section{Modo de Usuário, Modo de Supervisor e Chamadas de Sistema}
%% 

Um forte isolamento requer uma fronteira rígida entre aplicativos e o sistema operacional. Se o aplicativo cometer um erro, não queremos que o sistema operacional falhe ou que outros aplicativos falhem. Em vez disso, o sistema operacional deve ser capaz de limpar o aplicativo que falhou e continuar a executar outros aplicativos. Para alcançar um forte isolamento, o sistema operacional deve garantir que os aplicativos não podem modificar (ou mesmo ler) as estruturas de dados e instruções do sistema operacional e que os aplicativos não podem acessar a memória de outros processos.

As CPUs fornecem suporte de hardware para um forte isolamento. Por exemplo, o RISC-V tem três modos nos quais a CPU pode executar instruções: 
\indextext{modo máquina},
\indextext{ modo supervisor}, e
\indextext{modo usuário}.
Instruções executadas em modo máquina têm privilégio total; uma CPU começa em modo máquina. O modo máquina é principalmente destinado a configurar o computador durante a inicialização. O xv6 executa algumas linhas em modo máquina e depois muda para o modo supervisor.

No modo supervisor, a CPU pode executar \indextext{instruções privilegiadas}: por exemplo, habilitar e desabilitar interrupções, ler e escrever o registrador que contém o endereço de uma tabela de páginas, etc. Se um aplicativo em modo usuário tentar executar uma instrução privilegiada, a CPU não executa a instrução, mas muda para o modo supervisor para que o código em modo supervisor possa encerrar o aplicativo, porque ele fez algo que não deveria estar fazendo.
A Figura~\ref{fig:os} no Capítulo~\ref{CH:UNIX} illustrates this organization.  An application can
ilustra essa organização. Um aplicativo pode executar apenas instruções em modo usuário (por exemplo, somar números, etc.) e é dito estar rodando em espaço do usuário, enquanto o software em modo supervisor pode também executar instruções privilegiadas e é dito estar rodando em 
\indextext{espaço do usuário},enquanto o software em modo supervisor pode também executar instruções privilegiadas e é dito estar rodando em\indextext{espaço do kernel}. O software que roda em espaço do kernel (ou em modo supervisor) é chamado de\indextext{kernel}.

Um aplicativo que deseja invocar uma função do kernel (por exemplo, a chamada de sistema\lstinline{read} no xv6) deve transitar para o kernel; um aplicativo \emph{não pode}  invocar uma função do kernel diretamente. As CPUs fornecem uma instrução especial que muda a CPU do modo usuário para o modo supervisor e entra no kernel em um ponto de entrada especificado pelo kernel. (O RISC-V fornece a instrução\indexcode{ecall} para esse propósito.) Uma vez que a CPU mudou para o modo supervisor, o kernel pode então validar os argumentos da chamada de sistema (por exemplo, verificar se o endereço passado para a chamada de sistema é parte da memória do aplicativo), decidir se o aplicativo tem permissão para realizar a operação solicitada (por exemplo, verificar se o aplicativo tem permissão para escrever no arquivo especificado) e depois negar ou executar. É importante que o kernel controle o ponto de entrada para transições para o modo supervisor; se o aplicativo pudesse decidir o ponto de entrada do kernel, um aplicativo malicioso poderia, por exemplo, entrar no kernel em um ponto onde a validação dos argumentos é pulada.

\section{Organização do Kernel}
%% 

Uma pergunta chave de design é qual parte do sistema operacional deve rodar em modo supervisor. Uma possibilidade é que todo o sistema operacional resida no kernel, de modo que as implementações de todas as chamadas de sistema rodem em modo supervisor. Essa organização é chamada de
\indextext{kernel monolítico}.

Nesta organização, todo o sistema operacional consiste em um único programa rodando com privilégio total de hardware. Essa organização é conveniente porque o designer do OS não precisa decidir quais partes do sistema operacional não precisam de privilégio total de hardware. Além disso, é mais fácil para diferentes partes do sistema operacional cooperarem. Por exemplo, um sistema operacional pode ter um cache de buffer que pode ser compartilhado tanto pelo sistema de arquivos quanto pelo sistema de memória virtual.

Uma desvantagem da organização monolítica é que as interações entre diferentes partes do sistema operacional costumam ser complexas (como veremos no resto deste texto) e, portanto, é fácil para um desenvolvedor de sistema operacional cometer um erro. Em um kernel monolítico, um erro é fatal, pois um erro em modo supervisor frequentemente fará com que o kernel falhe. Se o kernel falhar, o computador para de funcionar, e assim todos os aplicativos falham também. O computador deve reiniciar para começar novamente.

Para reduzir o risco de erros no kernel, os designers de sistemas operacionais podem minimizar a quantidade de código do sistema operacional que roda em modo supervisor e executar a maior parte do sistema operacional em modo usuário. Essa organização do kernel é chamada de
\indextext{microkernel}.

\begin{figure}[t]
\center
\includegraphics[scale=0.5]{fig/mkernel.pdf}
\caption{Um microkernel com um servidor de sistema de arquivor}
\label{fig:mkernel}
\end{figure}

A Figura~\ref{fig:mkernel}ilustra esse design de microkernel. Na figura, o sistema de arquivos roda como um processo de nível de usuário. Os serviços do sistema operacional que rodam como processos são chamados de servidores. Para permitir que os aplicativos interajam com o servidor de arquivos, o kernel fornece um mecanismo de comunicação entre processos para enviar mensagens de um processo de modo usuário para outro. Por exemplo, se um aplicativo como o shell quiser ler ou escrever um arquivo, ele envia uma mensagem ao servidor de arquivos e aguarda uma resposta.

Em um microkernel, a interface do kernel consiste em algumas funções de baixo nível para iniciar aplicativos, enviar mensagens, acessar hardware de dispositivos, etc. Essa organização permite que o kernel seja relativamente simples, já que a maior parte do sistema operacional reside em servidores de nível de usuário.

Na prática, tanto os kernels monolíticos quanto os microkernels são populares. Muitos kernels Unix são monolíticos. Por exemplo, o Linux tem um kernel monolítico, embora algumas funções do OS rodem como servidores de nível de usuário (por exemplo, o sistema de janelas). O Linux oferece alto desempenho para aplicativos intensivos em OS, em parte porque os subsistemas do kernel podem ser integrados de forma estreita.

Sistemas operacionais como Minix, L4 e QNX são organizados como um microkernel com servidores e têm visto ampla implantação em configurações embarcadas. Uma variante do L4, seL4, é pequena o suficiente para ter sido verificada quanto à segurança de memória e outras propriedades de segurança~\cite{sel4}.

Há muito debate entre os desenvolvedores de sistemas operacionais sobre qual organização é melhor, e não há evidências conclusivas de um lado ou de outro. Além disso, isso depende muito do que significa "melhor": desempenho mais rápido, menor tamanho de código, confiabilidade do kernel, confiabilidade do sistema operacional completo (incluindo serviços de nível de usuário), etc.

Existem também considerações práticas que podem ser mais importantes do que a questão de qual organização. Alguns sistemas operacionais têm um microkernel, mas executam alguns dos serviços de nível de usuário em espaço do kernel por razões de desempenho. Alguns sistemas operacionais têm kernels monolíticos porque foi assim que começaram e há pouco incentivo para mudar para uma organização de microkernel puro, pois novas funcionalidades podem ser mais importantes do que reescrever o sistema operacional existente para se ajustar a um design de microkernel.

Do ponto de vista deste livro, sistemas operacionais microkernel e monolíticos compartilham muitas ideias-chave. Eles implementam chamadas de sistema, usam tabelas de páginas, lidam com interrupções, suportam processos, usam travas para controle de concorrência, implementam um sistema de arquivos, etc. Este livro foca nessas ideias centrais.

O xv6 é implementado como um kernel monolítico, como a maioria dos sistemas operacionais Unix. Assim, a interface do kernel xv6 corresponde à interface do sistema operacional, e o kernel implementa o sistema operacional completo. Como o xv6 não fornece muitos serviços, seu kernel é menor do que alguns microkernels, mas conceitualmente o xv6 é monolítico.

\section{Visão Geral do Processo}
%% 

A unidade de isolamento no xv6 (assim como em outros sistemas operacionais Unix) é um
\indextext{processo}.A abstração de processo impede que um processo destrua ou espione a memória, CPU, descritores de arquivo, etc. de outro processo. Também impede que um processo destrua o próprio kernel, de modo que um processo não possa subverter os mecanismos de isolamento do kernel. O kernel deve implementar a abstração de processo com cuidado, pois um aplicativo com bugs ou malicioso pode enganar o kernel ou o hardware para fazer algo ruim (por exemplo, contornar o isolamento). Os mecanismos usados pelo kernel para implementar processos incluem a bandeira de modo usuário/supervisor, espaços de endereço e fatiamento de tempo de threads.

Para ajudar a impor o isolamento, a abstração de processo fornece a ilusão a um programa de que ele tem sua própria máquina privada. Um processo fornece a um programa o que parece ser um sistema de memória privado, ou\indextext{espaço de endereço},  que outros processos não podem ler ou escrever. Um processo também fornece ao programa o que parece ser sua própria CPU para executar as instruções do programa.

O xv6 usa tabelas de páginas (que são implementadas pelo hardware) para dar a cada processo seu próprio espaço de endereço. A tabela de páginas do RISC-V traduz (ou "mapeia") um \indextext{endereço virtual}(o endereço que uma instrução RISC-V manipula) para um\indextext{endereço físico} (um endereço que a CPU envia para a memória principal).

\begin{figure}[t]
\centering
\includegraphics[scale=0.5]{fig/as.pdf}
\caption{Layout do espaço de endereço virtual de um processo}
\label{fig:as}
\end{figure}

O xv6 mantém uma tabela de páginas separada para cada processo que define o espaço de endereço desse processo. Como ilustrado na Figura~\ref{fig:as},um espaço de endereço inclui a\indextext{memória do usuário}do processo começando no endereço virtual zero. As instruções vêm primeiro, seguidas por variáveis globais, depois a pilha, e finalmente uma área de "heap" (para malloc) que o processo pode expandir conforme necessário. Existem vários fatores que limitam o tamanho máximo do espaço de endereço de um processo:
Os ponteiros no RISC-V têm 64 bits de largura; o hardware usa apenas os 39 bits mais baixos ao procurar endereços virtuais nas tabelas de páginas; e o xv6 usa apenas 38 desses 39 bits. Assim, o endereço máximo é$2^{38}-1$ =
0x3fffffffff, que é\lstinline{MAXVA}~\lineref{kernel/riscv.h:/define.MAXVA/}.
No topo do espaço de endereço, o xv6 coloca uma página de 
 \indextext{trampolim } paginá (4096 bytes)
e uma página de \indextext{trapframe}. O xv6 usa essas duas páginas para transitar para o kernel e de volta; a página de trampolim contém o código para fazer a transição para dentro e para fora do kernel, e o trapframe é onde o kernel salva os registradores de usuário do processo, como explicado no Capítulo\ref{CH:TRAP}.

O kernel xv6 mantém muitos pedaços de estado para cada processo, que reúne em uma
\indexcode{struct proc}
\lineref{kernel/proc.h:/^struct.proc/}.
 Os pedaços mais importantes do estado do kernel de um processo são sua tabela de páginas, sua pilha de kernel e seu estado de execução.
Usaremos a notação
\indexcode{p->xxx}
para nos referir a elementos da 
\lstinline{proc}
estrutura; por exemplo,
\indexcode{p->pagetable} é um ponteiro para a tabela de páginas do processo.

Cada processo tem um thread de controle (ou \indextext{thread}para abreviar) que mantém o estado necessário para executar o processo. Em um dado momento, um thread pode estar executando em uma CPU ou suspenso (não executando, mas capaz de retomar a execução no futuro). Para alternar uma CPU entre processos, o kernel suspende o thread que está atualmente rodando naquela CPU e salva seu estado, e restaura o estado de outro thread previamente suspenso de um processo. Grande parte do estado de um thread (variáveis locais, endereços de retorno de chamadas de função) é armazenada nas pilhas do thread. Cada processo tem duas pilhas: uma pilha de usuário e uma pilha de kernel (\indexcode{p->kstack}).
Quando o processo está executando instruções de usuário, apenas sua pilha de usuário está em uso, e sua pilha de kernel está vazia. Quando o processo entra no kernel (para uma chamada de sistema ou interrupção), o código do kernel é executado na pilha de kernel do processo; enquanto um processo está no kernel, sua pilha de usuário ainda contém dados salvos, mas não está sendo usada ativamente. O thread de um processo alterna entre usar ativamente sua pilha de usuário e sua pilha de kernel. A pilha de kernel é separada (e protegida do código de usuário) para que o kernel possa executar mesmo se um processo tiver danificado sua pilha de usuário.

Um processo pode fazer uma chamada de sistema executando a instrução \indexcode{ecall}
do RISC-V. Essa instrução eleva o nível de privilégio do hardware e altera o contador de programa para um ponto de entrada definido pelo kernel. O código nesse ponto de entrada muda para a pilha de kernel do processo e executa as instruções do kernel que implementam a chamada de sistema. Quando a chamada de sistema é concluída, o kernel muda de volta para a pilha de usuário e retorna ao espaço do usuário chamando a instrução \indexcode{sret} que reduz o nível de privilégio do hardware e retoma a execução das instruções de usuário logo após a instrução de chamada de sistema. O thread de um processo pode "bloquear" no kernel para esperar por E/S e retomar de onde parou quando a E/S estiver concluída..

\indexcode{p->state} 
indica se o processo está alocado, pronto para rodar, atualmente em execução em uma CPU, esperando por E/S ou saindo.

\indexcode{p->pagetable}
 contém a tabela de páginas do processo, no formato que o hardware RISC-V espera. O xv6 faz com que o hardware de paginação use a
\lstinline{p->pagetable}
de um processo ao executar esse processo no espaço do usuário. A tabela de páginas de um processo também serve como registro dos endereços das páginas físicas alocadas para armazenar a memória do processo.

Em resumo, um processo agrupa duas ideias de design: um espaço de endereço para dar ao processo a ilusão de sua própria memória, e um thread para dar ao processo a ilusão de sua própria CPU. No xv6, um processo consiste em um espaço de endereço e um thread. Em sistemas operacionais reais, um processo pode ter mais de um thread para aproveitar múltiplas CPUs.

\texttt{sp}
com o endereço
\lstinline{stack0+4096},
o topo da pilha, porque a pilha no RISC-V cresce para baixo. Agora que o kernel tem uma pilha,
\lstinline{_entry}
chama o código C em
\lstinline{start}
\lineref{kernel/start.c:/^start/}.

A função
\lstinline{start}
 realiza algumas configurações que são permitidas apenas em modo máquina e, em seguida, muda para o modo supervisor. Para entrar no modo supervisor, o RISC-V fornece a instrução
\lstinline{mret}.
Essa instrução é usada com mais frequência para retornar de uma chamada anterior do modo supervisor para o modo máquina. A
\lstinline{start} não está retornando de tal chamada, mas configura as coisas como se estivesse: ela define o modo de privilégio anterior como supervisor no registrador
\lstinline{mstatus},
define o endereço de retorno como
\lstinline{main}
 escrevendo o endereço de
\lstinline{main}'s
no registrador
\lstinline{mepc},
 desabilita a tradução de endereços virtuais em modo supervisor escrevendo
\lstinline{0}
no registrador de tabela de páginas
\lstinline{satp},
 e delega todas as interrupções e exceções para o modo supervisor.

Antes de pular para o modo supervisor, 
\lstinline{start}
realiza mais uma tarefa: programa o chip do relógio para gerar interrupções de temporizador. Com essa organização feita,
\lstinline{start}
"retorna" ao modo supervisor chamando
\lstinline{mret}.
Isso faz com que o contador de programa mude para
\lstinline{main}
\lineref{kernel/main.c:/^main/},
o endereço previamente armazenado em  \lstinline{mepc}.

Após 
\lstinline{main}
\lineref{kernel/main.c:/^main/}  
inicializar vários dispositivos e subsistemas, ele cria o primeiro processo chamando 
\lstinline{userinit}
\lineref{kernel/proc.c:/^userinit/}.
O primeiro processo executa um pequeno programa escrito em assembly RISC-V, que faz a primeira chamada de sistema no xv6.
\indexcode{initcode.S} 
\lineref{user/initcode.S:3} carrega o número para a \lstinline{exec}
chamada de sistema, \lstinline{SYS_EXEC}
\lineref{kernel/syscall.h:/exec/},
 no registrador {\tt a7},
 e então chama \lstinline{ecall} para reentrar no kernel.

O kernel usa o número no registrador {\tt a7}  na \lstinline{syscall}
\lineref{kernel/syscall.c:/^syscall/} para chamar a chamada de sistema desejada. A tabela de chamadas de sistema \lineref{kernel/syscall.c:/syscalls/} mapeia
\lstinline{SYS_EXEC} para a função \lstinline{sys_exec},que o kernel invoca.   Como vimos no Capítulo~\ref{CH:UNIX}, \indexcode{exec}
substitui a memória e os registradores do processo atual por um novo programa (neste caso, \indexcode{/init}).

Uma vez que o kernel completou a
\lstinline{exec},

\lstinline{init}
\lineref{user/init.c:/^main/}
cria um novo arquivo de dispositivo de console, se necessário, e então o abre como descritores de arquivo 0, 1 e 2. Em seguida, ele inicia um shell no console. O sistema está em funcionamento.

\section{Modelo de Segurança}

Você pode se perguntar como o sistema operacional lida com código com bugs ou malicioso. Como lidar com a malícia é estritamente mais difícil do que lidar com bugs acidentais, é razoável focar principalmente em fornecer segurança contra a malícia. Aqui está uma visão geral de alto nível das suposições e objetivos típicos de segurança no design de sistemas operacionais.

O sistema operacional deve assumir que o código de nível de usuário de um processo fará o possível para prejudicar o kernel ou outros processos. O código do usuário pode tentar desreferenciar ponteiros fora de seu espaço de endereço permitido; pode tentar executar qualquer instrução RISC-V, mesmo aquelas não destinadas ao código do usuário; pode tentar ler e escrever qualquer registrador de controle RISC-V; pode tentar acessar diretamente o hardware dos dispositivos; e pode passar valores engenhosos para chamadas de sistema na tentativa de enganar o kernel para que ele falhe ou faça algo imprudente. O objetivo do kernel é restringir cada processo de usuário para que tudo o que ele possa fazer é ler/escrever/executar sua própria memória de usuário, usar os 32 registradores de propósito geral do RISC-V e afetar o kernel e outros processos das maneiras que as chamadas de sistema permitem. O kernel deve prevenir quaisquer outras ações. Isso é tipicamente um requisito absoluto no design do kernel.

As expectativas para o próprio código do kernel são bastante diferentes. O código do kernel é presumido como sendo escrito por programadores bem-intencionados e cuidadosos. Espera-se que o código do kernel seja livre de bugs e, certamente, que não contenha nada malicioso. Essa suposição afeta a forma como analisamos o código do kernel. Por exemplo, existem muitas funções internas do kernel (por exemplo, as travas de spin) que causariam problemas sérios se o código do kernel as utilizasse incorretamente. Ao examinar qualquer parte específica do código do kernel, queremos nos convencer de que ele se comporta corretamente. Presumimos, no entanto, que o código do kernel, em geral, é escrito corretamente e segue todas as regras sobre o uso das próprias funções e estruturas de dados do kernel. Em nível de hardware, presume-se que a CPU RISC-V, RAM, disco, etc., operem conforme anunciado na documentação, sem bugs de hardware.

Claro, na vida real as coisas não são tão simples. É difícil impedir que código de usuário engenhoso torne um sistema inutilizável (ou faça com que ele entre em pânico) ao consumir recursos protegidos pelo kernel — espaço em disco, tempo de CPU, slots da tabela de processos, etc. É geralmente impossível escrever código de kernel livre de bugs ou projetar hardware livre de bugs; se os autores de código de usuário malicioso estiverem cientes de bugs no kernel ou no hardware, eles os explorarão. Mesmo em kernels maduros e amplamente utilizados, como o Linux, as pessoas descobrem continuamente novas vulnerabilidades~\cite{mitre:cves}. Vale a pena projetar salvaguardas no kernel contra a possibilidade de que ele tenha bugs: asserções, verificação de tipos, páginas de proteção de pilha, etc. Finalmente, a distinção entre código de usuário e código de kernel às vezes é borrada: alguns processos de nível de usuário privilegiados podem fornecer serviços essenciais e efetivamente fazer parte do sistema operacional, e em alguns sistemas operacionais, código de usuário privilegiado pode inserir novo código no kernel (como nos módulos de kernel carregáveis do Linux).

\section{Mundo Real}
%% 

A maioria dos sistemas operacionais adotou o conceito de processo, e a maioria dos processos se assemelha aos do xv6. No entanto, sistemas operacionais modernos suportam várias threads dentro de um processo, permitindo que um único processo aproveite múltiplas CPUs. O suporte a múltiplas threads em um processo envolve uma boa quantidade de maquinário que o xv6 não possui, frequentemente incluindo mudanças na interface (por exemplo, o
\lstinline{clone},
do Linux, uma variante do 
\lstinline{fork}),
para controlar quais aspectos de um processo as threads compartilham.
